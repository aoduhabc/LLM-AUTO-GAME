# 基于 LLM 自动生成内容的乡村探索小游戏开发设计文档

## 1. 项目概述

### 1.1 项目名称

**《乡野絮语》**（暂定）

### 1.2 项目定位

一款基于大语言模型（LLM）驱动的轻量级 2D 探索小游戏。玩家在乡村风格地图中自由探索，当进入新区域时，系统自动生成场景对应的 NPC，并调用 LLM 为 NPC 生成身份设定、性格特征和实时对话回复。玩家与 NPC 的交流内容由 LLM 动态生成，形成具有随机性、沉浸感和连续记忆的互动体验。

### 1.3 项目目标

本项目旨在实现以下体验目标：

1. 玩家探索新地图时，系统自动生成简单 NPC。
2. NPC 的背景、说话风格、回复内容由 LLM 自动生成。
3. 整体地图具备优美乡村风格，营造轻松治愈氛围。
4. 玩家与 NPC 的交流内容统一由 LLM 驱动生成。
5. 项目采用 React 前端实现主要交互与画面逻辑，后端采用 Python 实现生成服务与游戏状态服务。

---

## 2. 核心玩法设计

## 2.1 玩法概述

玩家扮演一位旅人，在一个乡村世界中漫步探索。地图按区域划分，例如：

* 村口小路
* 麦田边缘
* 河流木桥
* 果园
* 老旧风车旁
* 村民集市
* 山脚茶棚

当玩家首次进入某个未探索区域时，系统会触发以下流程：

1. 根据当前地图区域类型生成一名或多名 NPC。
2. 为 NPC 生成基础资料，如名字、职业、性格、当前状态。
3. 生成 NPC 的初始欢迎语。
4. 玩家可与 NPC 发起自由对话或选择系统推荐话题。
5. 后端调用 LLM 生成 NPC 回复，并可根据对话上下文持续保持人物一致性。

---

## 2.2 核心循环

游戏主循环如下：

**探索地图 → 发现新区域 → 自动生成 NPC → 发起对话 → 获得线索/故事/情绪反馈 → 继续探索**

该循环强调“轻探索 + 轻叙事 + 高随机对话”的体验，而非复杂战斗或高强度任务驱动。

---

## 2.3 玩家行为

玩家可进行的主要行为包括：

* 移动探索
* 与 NPC 对话
* 查看已认识 NPC 档案
* 回顾历史对话
* 触发区域小事件
* 收集关键词、传闻、故事碎片

后续可扩展：

* NPC 好感度
* 天气/昼夜变化
* 小型任务链
* 物品收集与交换
* 玩家自定义提问风格

## 3. 美术与风格设计

## 3.1 整体视觉风格

地图采用 **温暖、柔和、田园化** 的乡村视觉风格，整体偏治愈与自然感。推荐风格关键词如下：

* 乡村小道
* 稻田与麦浪
* 木栅栏与石桥
* 果树、野花、风车
* 暖色夕阳
* 低饱和自然配色

建议采用 **2D 俯视角 tile map** 方案，这样开发成本较低、资源复用高，也方便做探索式交互。

---

## 3.2 地图构成

地图由多个“区域块”拼接组成，每个区域块包含：

* 地形类型：草地、田埂、河边、院落、集市
* 装饰元素：树、花丛、木桶、石灯、农具
* 可交互点：NPC 出现点、剧情触发点、休息点

地图生成上不建议完全实时由 LLM 生成图像，而建议采用以下混合方案：

### 推荐方案

* **地图结构**：程序化生成或模板拼接
* **地图美术资源**：预先设计好的乡村风素材包
* **区域文本描述**：由 LLM 辅助生成，用于增强叙事氛围

这样可以兼顾：

* 游戏运行性能
* 美术一致性
* 开发可控性
* LLM 使用成本

## 4. LLM 驱动设计

## 4.1 LLM 在本项目中的职责

LLM 不负责实时渲染画面，而主要承担以下内容生成任务：

1. **NPC 生成**

   * 名字
   * 年龄段
   * 职业
   * 性格标签
   * 当前心情
   * 所属区域
   * 背景故事摘要

2. **对话生成**

   * 首次见面打招呼
   * 对玩家问题的自然回复
   * 基于历史聊天的连续回应
   * 与地图区域风格一致的表达内容

3. **玩家候选话题生成**

   * 系统推荐可聊的话题
   * 引导玩家继续探索

4. **事件文本生成**

   * 场景描述
   * 传闻
   * 小故事
   * 区域氛围旁白

---

## 4.2 NPC 自动生成机制

### 触发时机

当玩家满足以下条件时触发 NPC 自动生成：

* 首次进入某地图区域
* 该区域当前无已生成 NPC
* 区域允许生成 NPC
* 服务端未缓存该区域的 NPC 数据

### 生成输入

输入给 LLM 的上下文包括：

* 区域类型（如麦田、桥边、集市）
* 当前时间（早晨/黄昏/夜晚）
* 世界整体风格（治愈乡村）
* NPC 数量限制
* 人物设定约束（简单、清晰、适合小游戏）

### 生成输出示例

```json
{
  "npc_id": "npc_001",
  "name": "阿禾",
  "role": "年轻农夫",
  "personality": ["温和", "有点腼腆", "喜欢观察天气"],
  "mood": "放松",
  "location": "麦田边",
  "background": "从小在村里长大，熟悉每一片田地。",
  "opening_line": "你也是来这里看风吹麦浪的吗？这会儿的景色最好。"
}
```

## 4.3 对话生成机制

### 对话原则

NPC 回复必须满足以下要求：

1. 保持角色一致性
2. 回答简洁自然，适合小游戏节奏
3. 不脱离乡村世界观
4. 能记住与玩家最近几轮对话
5. 避免长篇大论
6. 可产生轻微引导作用，如提示附近地点或人物

### 输入上下文

对话生成时需要提供：

* NPC 的基础设定
* 当前区域
* 最近几轮聊天记录
* 玩家输入
* 世界风格约束
* 回复长度约束

### 输出要求

LLM 返回内容建议包含：

* `reply_text`：NPC 回复文本
* `emotion`：当前情绪标签
* `suggestions`：可推荐给玩家的后续话题
* `memory_update`：需要写入 NPC 短期记忆的摘要

---

## 4.4 对话记忆机制

为了让 NPC 具备“持续认识玩家”的感觉，建议设计两层记忆：

### 短期记忆

保存最近 5~10 轮对话，用于当前会话连续性。

### 长期记忆

保存关键事实，例如：

* 玩家曾问过什么
* 玩家提过喜欢的地方
* 玩家帮 NPC 做过什么
* 玩家与该 NPC 的关系状态

长期记忆不宜过长，可通过摘要方式保存，避免 token 消耗过高。

## 8. 核心业务流程

## 8.1 首次进入新区域流程

1. 玩家进入区域触发点。
2. 前端请求后端查询该区域是否已生成 NPC。
3. 若未生成，则后端根据区域信息调用 NPC 生成服务。
4. 后端将生成结果写入数据库。
5. 前端在地图中渲染 NPC。
6. NPC 头顶显示简短引导文本。

---

## 8.2 玩家发起对话流程

1. 玩家靠近 NPC 并点击交互。
2. 前端打开对话窗口。
3. 前端请求后端获取该 NPC 初始上下文。
4. 玩家输入文字或点击推荐话题。
5. 后端组装 prompt 并调用 LLM。
6. 返回 NPC 回复文本和推荐后续话题。
7. 后端更新对话历史与记忆摘要。
8. 前端继续展示后续互动。

---

## 8.3 NPC 连续记忆流程

1. 对话结束后，将关键事实写入长期记忆。
2. 下次再次见到该 NPC 时，后端优先读取长期记忆。
3. LLM 在生成欢迎语时可体现“认出玩家”的效果。

例如：

* “你上次提到想去桥那边看看，后来去了吗？”

## 10. API 设计

## 10.1 区域探索接口

### `POST /api/region/enter`

请求参数：

```json
{
  "player_id": "player_001",
  "region_id": "region_wheat_01"
}
```

返回参数：

```json
{
  "first_visit": true,
  "npc_generated": true,
  "npcs": [
    {
      "npc_id": "npc_001",
      "name": "阿禾",
      "role": "年轻农夫",
      "position": {"x": 12, "y": 8},
      "opening_line": "你也是来这里看风吹麦浪的吗？"
    }
  ]
}
```

---

## 10.2 发起对话接口

### `POST /api/dialogue/chat`

请求参数：

```json
{
  "player_id": "player_001",
  "npc_id": "npc_001",
  "message": "这里平时一直这么安静吗？"
}
```

返回参数：

```json
{
  "reply_text": "大多数时候是的，只有风吹过麦子的时候，会像有人在轻声说话。",
  "emotion": "平静",
  "suggestions": [
    "这附近还有什么值得去的地方？",
    "你每天都会来这里吗？"
  ]
}
```

## 11. Prompt 设计思路

## 11.1 NPC 生成人设 Prompt

核心约束：

* 角色适合乡村世界
* 设定简单清楚
* 语气自然
* 不要生成复杂设定或超自然内容
* 输出 JSON 结构

示例思路：

* 输入区域类型：麦田边
* 输出角色：年轻农夫、拾穗老人、送牛奶的姑娘等

---

## 11.2 对话 Prompt

核心约束：

* 你是某 NPC
* 你只知道乡村世界内的信息
* 回复不超过 50 字或 80 字
* 语气符合人物设定
* 可温和引导探索
* 不要脱离角色回答技术性问题

---

## 11.3 兜底策略

当 LLM 输出异常时，启用默认回复，例如：

* “今天天气真好，适合慢慢走走。”
* “这事我一下也说不清，不如你去前面再看看。”
* “村里每天都有新鲜事，你多转转就知道了。”

---

## 12. 性能与工程约束

## 12.1 延迟控制

LLM 调用可能成为体验瓶颈，因此需要优化：

1. 首次区域进入时，NPC 生成可提前加载。
2. 对话回复支持流式输出。
3. 常见场景可缓存 NPC 人设。
4. 常见打招呼语可预生成。
5. 历史上下文做摘要，减少 token 长度。

---

## 12.2 成本控制

建议不要让 LLM 参与所有游戏逻辑，只负责高价值文本生成。

适合调用 LLM 的时机：

* 生成 NPC
* 首次对话
* 关键互动
* 有历史记忆的连续对话

不适合调用 LLM 的时机：

* 角色移动
* 地图碰撞
* 普通渲染
* 每帧更新
* 简单按钮逻辑

---

## 12.3 稳定性要求

需要加入以下机制：

* 超时重试
* 接口限流
* 输出校验
* JSON 解析失败兜底
* 敏感内容过滤
* 非法输入拦截

## 15. 风险与应对

## 风险 1：LLM 回复不稳定

应对：

* 强约束 prompt
* 输出 JSON 化
* 加兜底模板

## 风险 2：响应速度慢

应对：

* 缓存
* 流式输出
* 控制上下文长度

## 风险 3：NPC 设定前后不一致

应对：

* 固化基础人设
* 长期记忆摘要
* 每次对话都注入角色卡

## 风险 4：地图生成和美术风格不统一

应对：

* 使用固定素材包
* LLM 只负责文本，不负责实时视觉生成

---

## 16. 验收标准

项目达到 MVP 验收时，应满足：

1. 玩家可以在乡村风格地图中自由移动。
2. 玩家进入新区域时，能自动生成至少 1 个 NPC。
3. NPC 能以符合自身设定的风格与玩家自然对话。
4. 对话支持连续上下文，不是完全随机断裂回复。
5. 前端主要功能基于 React 实现，后端使用 Python。
6. 整体游戏体验完整、可演示、可继续扩展。

---

## 17. 推荐的落地实现方案

为了尽快做出一个“能跑、能看、能演示”的版本，推荐采用以下方案：

### 前端

* React + TypeScript
* Phaser 3
* Zustand
* Tailwind CSS

### 后端

* Python + FastAPI
* SQLite（MVP 阶段）
* Redis（可选）
* LLM API

### 生成策略

* 地图：模板拼接
* NPC：LLM 生成结构化角色卡
* 对话：LLM 动态生成
* 记忆：数据库存摘要

这个方案最适合小游戏原型开发，因为：

* 前端交互和画面可快速成型
* 后端逻辑清晰
* LLM 能充分体现产品亮点
* 后续容易扩展成更完整的 AI NPC 叙事游戏
